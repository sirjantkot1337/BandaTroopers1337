# RTO Support: дизайн интерфейса

## 1. Главный UX-принцип

Игрок должен без чтения кода понимать три вещи:

1. какие пакеты сейчас выбраны;
2. почему кнопка доступна или недоступна;
3. что произойдет после `Ctrl+Click` через бинокль.

## 2. Когда видны RTO-кнопки

RTO HUD показывается только если:

- у владельца есть `RTO binoculars` в левой или правой руке;
- владелец жив;
- владелец является `JOB_SQUAD_RTO`.

Если бинокль убран:

- кнопки не сереют;
- кнопки именно скрываются.

Это касается и `Выбрать пакеты поддержки`, и utility-режимов, и support abilities.

## 3. Preset selection

Меню выбора пакетов остается отдельным TGUI.

Кнопка входа:

- `Выбрать пакеты поддержки`

Кнопка должна оставаться доступной всегда, пока:

- support не отключен Game Rule Panel;
- у роли есть доступ к `rto_support`;
- бинокль в руке и HUD видим.

Меню должно явно показывать:

- `Slot 1`
- `Slot 2`
- уже выбранные пакеты
- countdown до полного reset
- возможность полного reset обоих слотов
- solo-бонус на cooldown сектора для одиночного боевого пакета

Игрок не должен угадывать, что второй пакет влияет на cooldown сектора. Это нужно писать в интерфейсе явно.

## 4. Что показывать в карточке пакета

В карточке пакета должны читаться:

- роль пакета;
- нужен ли сектор;
- радиус, длительность и cooldown сектора;
- reduced solo cooldown сектора, если пакет поддерживает сектор;
- список support-ability;
- общие и личные cooldown'ы support-ability;
- ограничения по открытому небу и целевому тайлу.

`Logistics`, `Medical`, `Technical` и HALO utility-пакеты в UI явно помечаются как пакеты без сектора.

## 5. Порядок кнопок на HUD

Порядок должен быть стабильным:

1. `Выбрать пакеты поддержки`
2. кнопки секторов в порядке слотов
3. `Координаты`
4. `Лазерная отметка`
5. support-кнопки первого пакета
6. support-кнопки второго пакета

## 6. Visibility actions

У каждого выбранного zone-based пакета своя отдельная кнопка сектора.

Кнопка сектора показывает только состояние самого сектора:

- `Наведение`
- `Активна: Ns`
- `Общий КД: Ns`
- `Личный КД: Ns`
- `Готово`

Она не должна:

- показывать общий cooldown support-пакета;
- выглядеть так, будто делит cooldown с support ability;
- скрывать факт solo-бонуса в меню пакетов.

## 7. Support abilities

### Zone-based templates

Для `Mortar`, `CAS`, `Heavy Strike` support button ведет себя так:

1. если сектора нет или он не активен:
   primary label `Нет сектора`
2. если сектор активен и у ability есть cooldown:
   primary label показывает только таймер самой ability
3. если сектор активен и cooldown нет:
   primary label `Готово`

Shared и personal cooldown'ы не рисуются как secondary labels.

Если одновременно активны local и shared cooldown:

- на кнопке показывается больший из двух;
- при равенстве приоритет отдается local cooldown.

### No-zone templates

Для `Logistics`, `Medical`, `Technical`, `HALO Logistics`, `HALO Medical`, `HALO Technical` zone labels запрещены полностью.

Primary label:

- больший из `Общий КД: Ns` и `Личный КД: Ns`, если активны оба
- иначе `Личный КД: Ns`, если активен personal cooldown
- иначе `Общий КД: Ns`, если активен shared cooldown
- иначе `Готово`

## 8. Utility modes

### `Координаты`

- persistent mode;
- без cooldown;
- после клика остается включенным;
- в интерфейсе показывает только:
  - `Наведение`, если режим активен
  - `Готово`, если режим не активен

### `Лазерная отметка`

- persistent mode;
- без cooldown;
- после клика остается включенной;
- подсветка живет, пока сохраняется реальное ведение цели через бинокль.

Интерфейс показывает режим как `Наведение`, пока marker mode активен.

## 9. Binocular UX

Бинокль является финальной точкой interaction flow:

- action-кнопка выбирает режим;
- бинокль принимает цель через `Ctrl+Click` во время зума;
- `Ctrl+Click` по самому биноклю снимает текущий armed mode.

Examine у бинокля должен объяснять:

- выбранные пакеты;
- состояние активного сектора;
- текущий armed mode;
- активна ли ручная лазерная отметка;
- countdown до полного reset слотов;
- активен ли solo-бонус на сектор.

## 10. Ролевой UX

UI не должен обещать одинаковый набор пакетов всем RTO.

Текущий контракт:

- `USCM`: `mortar`, `cas`, `heavy`, `logistics`, `medical`, `technical`
- `UNSC`: `mortar`, `halo_logistics`, `halo_medical`, `halo_technical`
- `ODST`: `cas`, `heavy`, `halo_logistics`, `halo_medical`, `halo_technical`

Именно доступные пакеты должны попадать в preset menu конкретного игрока.
