# SS220: Overlay над `.AI_AGENT`

Канонический entrypoint для AI-агента находится в [`../../AGENTS.md`](../../AGENTS.md).
Stable agent rules живут в [`../__agents/.AI_AGENT/`](../__agents/.AI_AGENT/README.md).
Этот документ больше не является общим глобальным сводом правил: он содержит только BandaTroopers/SS220-specific deltas для модульности, сопровождения апстрима и применения `SS220 EDIT`.

## 1. Матрица путей

### `modular/**`
- Основное место для новой бизнес-логики.
- Здесь живут новые подсистемы, расширения и большая часть адаптеров BandaTroopers.
- Новые `SS220 EDIT` внутри `modular/**` по умолчанию не добавляются.
- Уже существующие `SS220 EDIT` в `modular/**` считаются legacy-исключениями и не задают новый стандарт.

### `code/**` и другие пути вне `modular/**`
- Это апстрим-хардкод и интеграционный слой.
- Сюда вносятся только минимальные точки интеграции:
  - вызовы модульных хуков,
  - адаптеры,
  - безопасные fallback-ветки,
  - неизбежный glue-код совместимости.
- Любая крупная бизнес-логика должна оставаться в `modular/**`, если это технически возможно.

### `maps/**` и `map_config/**`
- Изменения карт, JSON-конфигов и ignore-списков должны рассматриваться совместно.
- Если задача меняет тестовую матрицу, ротацию или набор карт, нужно проверить связанные файлы вроде `.github/maps_to_ignore.txt`.
- Для map-sensitive изменений использовать релевантные map/build-проверки из `.AI_AGENT/WORKFLOW_RULES.md`.
- Для `map_config/**` и других согласованных config surfaces `SS220 EDIT` допускается, если это часть сопровождаемого апстрим-диффа.

### Docs и configs
- Документация и конфиги должны оставаться в UTF-8 и без mojibake.
- Не дублировать здесь общие agent-процессы, уже описанные в `.AI_AGENT`, если правило не является SS220-specific.

## 2. `SS220 EDIT` для апстрим-хардкода

Область действия:
- `code/**` как основной слой сопровождаемого апстрим-диффа.
- `map_config/**` и другие согласованные config surfaces, если там ведется тот же сопровождаемый дифф.
- Existing markers в `modular/**` считаются legacy debt и подлежат инвентаризации, но не расширяются без крайней причины.

Правила:
1. Удаляемые строки не удаляются полностью, если сохраняется смысл сравнения со старой логикой.
2. Старая логика остается рядом в виде закомментированных строк там, где это помогает сопровождать апстрим-дифф.
3. Если изменена одна строка, на этой же строке ставится:
   - `// SS220 EDIT: <краткое описание>`
4. Для мелкой одноcтрочной правки используется inline-маркер без `START/END`.
5. Если изменено несколько соседних строк, блок оборачивается:
   - `// SS220 EDIT - START`
   - `// SS220 EDIT - END`
6. Если строка заменяется, старая строка по возможности остается сразу над новой в закомментированном виде.
7. Inline-маркер должен содержать короткое и конкретное описание изменения.
8. `SS220 EDIT` используется там, где реально ведется сопровождаемый апстрим/config-дифф; existing markers в `modular/**` не считать шаблоном для новых правок.

Пример одной строки:

```dm
value = new_value // SS220 EDIT: источник значения изменен old_value -> new_value
```

Пример с добавленным аргументом:

```dm
target_proc(arg1, arg2, extra_arg) // SS220 EDIT: добавлен extra_arg для latejoin-контекста
```

Пример блока:

```dm
// SS220 EDIT - START
// old_call(arg1, arg2)
new_call(arg1, arg2, arg3)
if(new_condition)
	handle_new_flow()
// SS220 EDIT - END
```

Рекомендуемая проверка:

```bash
rg -n "SS220 EDIT" code map_config
rg -n "SS220 EDIT" modular -g "!modular/__agents/**" -g "!modular/__docs/**"
```

Вторая команда нужна для инвентаризации legacy-маркеров в `modular/**`, а не как разрешение размечать ими новые модульные изменения.

## 3. Правила модульности и сопровождения апстрима

Этот раздел дополняет [`../__agents/.AI_AGENT/POLICIES.md`](../__agents/.AI_AGENT/POLICIES.md) и [`../__agents/.AI_AGENT/WORKFLOW_RULES.md`](../__agents/.AI_AGENT/WORKFLOW_RULES.md).

1. Новая бизнес-логика сначала реализуется в `modular/**`.
2. Перед правкой апстрим-хардкода проверяются существующие extension points, signals, wrappers и modular hooks.
3. Предпочтение отдается адаптерам и оберткам, а не переписыванию апстрим-подсистем.
4. Дифф в апстрим-файлах должен быть минимальным и без несвязанных рефакторингов.
5. Стабильные апстрим-контракты и ключи не ломаются без критической необходимости и явного scope.
6. После синка с апстримом перепроверяются все маркеры `SS220` и соответствующие точки интеграции.
7. Если модуль можно отключить без ломки апстрима, это предпочти тельный дизайн.
8. Если модуль и хардкод разделяют стабильные `#define`/string-id контракты, такие константы выносятся в `code/__DEFINES/bandamarines/...`, а не держатся только в `modular/_*.dm`.

## 4. SS220-specific чеклист перед сдачей

1. Новая логика живет в `modular/**`, а не размазана по апстриму без причины.
2. В апстриме оставлены только минимальные точки интеграции.
3. `SS220 EDIT` размечен только там, где это действительно требуется.
4. Удаленные или замененные строки апстрима сохранены в комментариях там, где это помогает сопровождать дифф.
5. Map/config изменения синхронизированы, если задача затрагивала карты или ротацию.
6. Дифф локальный, читаемый и без лишних изменений.
